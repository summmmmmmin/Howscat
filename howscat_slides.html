CatOwnershipService — 크로스 커팅 인가

모든 케어 서비스 메서드 진입부에서 호출. JWT subject(userId) ↔ cat.userId 비교 → 불일치 시 SecurityException → GlobalExceptionHandler가 403 반환.

// 서비스 공통 진입부 패턴
@Service
public class WeightService {
  public void save(Long catId, Integer userId, ...) {
    catOwnershipService.validate(catId, userId);
    // → 403 or 계속 실행
    ...
  }
}
SecurityConfig 핵심 설정
  • Stateless 세션 — JWT 기반, 서버 세션 없음
  • CSRF 비활성화 — Stateless API에 적합
  • CORS — env var 기반, allowCredentials=true
  • JwtFilter — Bean 등록 없이 수동 주입 (이중 등록 방지)
  • Permit-all — login · signup · refresh · Swagger

왜 이 구조인가

Stateless JWT는 서버에서 취소할 수 없다. 탈취된 토큰이 만료 전까지 유효한 문제를 Redis 블랙리스트로 해결. Access Token을 30분으로 짧게 유지해 블랙리스트 보관량도 최소화했다.

왜 Redis가 아닌 MySQL인가

AI 건강 요약은 6시간 단위 갱신. Redis의 sub-ms 응답이 필요하지 않다. MySQL PK 조회로 충분히 빠르고, Redis는 JWT 블랙리스트에 집중해 역할을 분리했다.
